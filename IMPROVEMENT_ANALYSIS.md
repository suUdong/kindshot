## 1. 🔴 Critical Issues (즉시 해결)

### 1.1 테스트 커버리지 부족
**현황**: 24개 소스 파일 vs 20개 테스트 파일 (83% 커버리지)
- ❌ `telegram_ops.py` 테스트 없음
- ❌ `runtime_artifacts.py` 테스트 없음
- ❌ `sd_notify.py` 테스트 없음
- ❌ `__main__.py` CLI 엔드투엔드 테스트 부재

**영향**: 배포 후 버그 발견, 운영 중단 위험
**해결책**:
- [ ] 누락된 모듈 테스트 작성 (3개 모듈)
- [ ] CLI 엔드투엔드 테스트 추가
- [ ] CI/CD에서 커버리지 체크 (최소 90%)

---

### 1.2 에러 처리 전략 불일관
**현황**: 
- `except` 패턴이 산발적 (JSONDecodeError, ValueError 등)
- 에러 로깅 없이 조용히 실패하는 경로 다수
- 복구 전략 미정의

**예시**:
```python
# collector.py
except json.JSONDecodeError:
    # 로깅 없이 진행?
    pass
```

**영향**: 데이터 손실, 디버깅 어려움
**해결책**:
- [ ] 커스텀 예외 클래스 정의 (KindShotError 계층)
- [ ] 모든 except에 구조화된 로깅 추가
- [ ] 복구 불가능한 에러는 fail-stop (현재 logger.py의 LogWriteError 패턴 확대)

---

### 1.3 LLM API 호출의 재시도 로직 부재
**현황**: `context_card.py`, `decision.py`, `unknown_review.py`에서 Anthropic API 호출
- ❌ 네트워크 에러 시 재시도 없음
- ❌ Rate limit 처리 미흡
- ❌ 타임아웃 설정 불명확

**영향**: 시장 기회 손실, 거래 지연
**해결책**:
- [ ] exponential backoff 재시도 래퍼 작성
- [ ] Rate limit 감지 및 자동 대기
- [ ] 타임아웃 설정 문서화 및 테스트

---

## 2. 🟠 High Priority (1-2주 내)

### 2.1 로깅 구조 개선
**현황**:
- JSONL 로깅은 좋지만, 구조화되지 않은 로그 메시지 혼재
- 로그 레벨(DEBUG, INFO, WARNING, ERROR) 사용 불일관
- 성능 추적 (polling_trace.py) vs 비즈니스 로그 분리 미흡

**영향**: 운영 분석 어려움, 성능 병목 파악 곤란
**해결책**:
- [ ] 로그 레벨 정책 수립 (DEBUG=개발용, INFO=운영용 등)
- [ ] 구조화 로깅 스키마 정의 (필드 표준화)
- [ ] 로그 집계 도구 통합 (ELK, Datadog 등)

---

### 2.2 설정 관리 개선
**현황**: `config.py`에서 환경변수 + 기본값
- ❌ 설정 검증 미흡 (예: API 키 존재 여부만 체크)
- ❌ 설정 변경 시 재시작 필요 (핫 리로드 불가)
- ❌ 설정 감시 기능 없음 (변경 감지 불가)

**영향**: 운영 중 설정 변경 어려움, 실수 위험
**해결책**:
- [ ] Pydantic 설정 검증 강화 (범위, 타입, 상호 의존성)
- [ ] 설정 파일 (YAML/TOML) 지원 추가
- [ ] 설정 변경 감시 및 로깅

---

### 2.3 메모리 누수 가능성
**현황**: `main.py`의 asyncio 이벤트 루프
- ❌ 장시간 실행 시 메모리 증가 추세 불명확
- ❌ 캐시 (decision.py의 LLM 캐시) 크기 제한 없음
- ❌ 연결 풀 관리 미흡 (aiohttp 세션)

**영향**: 서버 메모리 압박, OOM 위험
**해결책**:
- [ ] 메모리 프로파일링 (memory_profiler 추가)
- [ ] LLM 캐시 크기 제한 (LRU 캐시 적용)
- [ ] aiohttp 세션 풀 설정 검토

---

### 2.4 Unknown 신호 처리 파이프라인 미완성
**현황**: `unknown_review.py` 구현 중
- ❌ 일일 UNKNOWN 수집 로직 부분적
- ❌ 배치 LLM 분류 스케줄링 미정의
- ❌ 피드백 루프 (분류 결과 → 버킷 업데이트) 없음

**영향**: UNKNOWN 비중 높음 (실제 신호 누락)
**해결책**:
- [ ] 일일 UNKNOWN 수집 자동화 (cron 통합)
- [ ] 배치 LLM 분류 스케줄링 (야간 배치)
- [ ] 분류 결과 피드백 루프 구현

## 7. 💡 추가 고려사항

### 7.1 기술 부채
- `collector.py`의 백필 로직이 복잡 (리팩토링 필요)
- `main.py`의 asyncio 조율이 명시적이지 않음 (문서화 필요)
- `bucket.py`의 키워드 리스트가 하드코딩 (설정 파일화 필요)

### 7.2 확장성
- 현재 단일 종목 거래 → 다중 종목 확장 시 아키텍처 변경 필요
- 현재 일중거래 → 스윙/장기 거래 확장 시 데이터 모델 변경 필요
- 현재 뉴스 신호만 → 기술적 지표 추가 시 decision.py 확장 필요

### 7.3 규제 준수
- 거래 기록 보관 기간 미정의 (법적 요구사항 확인 필요)
- 감시 체계 (거래 모니터링) 미흡 (금감원 규정 확인 필요)
